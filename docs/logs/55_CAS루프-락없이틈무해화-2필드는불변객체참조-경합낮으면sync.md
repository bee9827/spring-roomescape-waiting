# 55. CAS 루프 — 틈을 없애는 게 아니라 무해하게 + 2필드는 불변 객체 참조로 + 경합 낮으면 sync가 이긴다

**날짜**: 2026-07-03 (같은 날 네 번째 사이클 — arc '동시성 손끝 증명' 1칸)
**학습 범위**: 새 hot arc '동시성 손끝 증명'의 #1 칸(compareAndSet) — 키워드 이해→흐름 파악→실전 판단까지 한 번에. 재방문: log_47(DB 낙관 CAS — 인출 출발점), log_52(synchronized·check-then-act·AtomicLong 탈락).

분류: DB·트랜잭션(동시성) — 흐름 파악 + 실전 판단

## 1. 출발은 인출 — log_47의 DB CAS가 이미 같은 뼈대
- `UPDATE ... SET status=새 WHERE status=기대` — 락 없이 안전했던 비결: **affected rows**. 0행 = "기대가 이미 낡음 = 누가 먼저 감"을 *결과로* 통보. 잠그고 확인이 아니라 **일단 지르고 결과로 판정**.
- 낙관의 정밀화: 낙관하는 건 "락이 안 발생"이 아니라 **"충돌이 드물다"**. 비관 = 충돌 흔함 가정 → 매번 잠그는 비용 / 낙관 = 충돌 드묾 가정 → 평소 공짜, 드문 충돌 때만 재시도 비용. 실패 후 행동(재시도/기권/에러)은 구현 나름 — log_47은 기권, 버킷은 재시도.
- `compareAndSet(기대값, 새값)` = 그 UPDATE 문을 **메모리 연산 하나**로 압축한 것. 반환 boolean = affected rows.

## 2. CAS 루프 — 순서가 전부
```java
while (true) {
    long current = tokens.get();                       // ① 읽기
    if (current < 1) return false;                     // ② 판정 — 반드시 CAS 앞!
    if (tokens.compareAndSet(current, current - 1)) {  // ③ 비교+변경 한 방
        return true;
    }
    // false → 내 정보가 낡았을 뿐. 기다림 0으로 즉시 재시도(경합은 나노초)
}
```
- **판정이 CAS 뒤면 터진다**: tokens=0에서 `CAS(0, -1)`은 *성공해버린다* — CAS는 "기대와 같으면 바꾼다"만 알아서. 위험한 건 실패하는 CAS가 아니라 **성공해버리는 CAS**.
- **두 가지 "다시"를 구분**: CAS false(경합, 정보 낡음) → 즉시 재시도, 공짜 / 읽은 값 <1(진짜 없음) → 거절(즉시 버전) 또는 여기서부터가 시간의 영역(블로킹 버전 maxWait). CAS 루프에 시계는 등장하지 않는다.
- ②와 ③ 사이 **틈은 여전히 존재**하지만 무해 — 끼어듦이 있었으면 ③이 거부하고 루프는 항상 신선한 값으로 재판정. **synchronized = 틈을 없앰(비관) / CAS = 틈을 무해화(낙관).**
- **AtomicLong 명예회복**: log_52에서 탈락한 이유("원자성은 연산 하나만, ①-② 사이 못 닫음")는 여전히 참이지만, `compareAndSet`은 비교와 변경이 붙은 *한 연산*이라 "사이"가 없다. 직감이 틀린 게 아니라 **루프 패턴 하나가 모자랐던 것**.

## 3. 필드가 두 개면 CAS가 막힌다 → 불변 객체 + AtomicReference
- "CAS 하고 나서 refill 하면?"(학습자 반격) → 문제가 사라지지 않고 **이사 간다**: refill 자체가 2필드(tokens, lastRefillNanos) check-then-act. 락 없이 동시 refill → 둘 다 같은 lastRefillNanos를 읽고 각자 +20 = **더블 보충**(한도 장치가 스스로 한도를 뻥튀기).
- 발상 전환: CAS가 변수 하나만 묶는다면 **두 필드를 변수 하나로** — 불변 record로 포장 + `AtomicReference<State>` CAS. old를 고치지 않고(고치면 도로 2필드 따로 쓰기) **새 상태를 만들어 참조를 통째로 교체**. 낡으면 거부→재계산. Bucket4j가 정확히 이 구조.

## 4. 실전 판단 — 우리 버킷은 synchronized로 남는다
- 학습자 1차 판단("경합 적으니 lock-free가 훨씬 좋다") → 두 질문으로 뒤집힘: ① 락 *대기*는 줄 섰을 때만 발생 — 경합 적으면 **락 비용 이미 ~0**, 없는 병 고치는 약. ② 코드 단순성: 키워드 하나 vs record+참조+재계산+루프.
- **깊은 층(학습자 발굴)**: "내가 쓰던 낙관/비관이랑 여기는 좀 다르네" → 저울(충돌 빈도)은 같고 **락의 가격표가 다르다**. DB 비관 락은 무경합에도 비쌈(왕복·트랜잭션 홀드·데드락) → "드물면 낙관"이 강한 규칙. JVM synchronized는 무경합이면 거의 공짜(uncontended fast path) → 드물면 아낄 게 없어 **결정 축이 단순성으로 이동**.
- 결론: **안 바꾼다.** Bucket4j는 반대 상황(수백 스레드 경합의 공용 라이브러리)의 답. **도구 선택은 이념이 아니라 경합 규모와 가격표.**

## 5. 재발명 3연속
bounded wait(→ j.u.c. `tryAcquire(timeout)` 관례, log_52) → 서킷 브레이커(→ Resilience4j 설정 1:1, log_54) → 버킷 API(→ Bucket4j `tryConsume`/`asBlocking().tryConsume(n, maxWait)` 1:1). *직접 도출한 설계가 곧 업계 표준 인터페이스*라는 확인이 사이클마다 보상으로 작동.

## 한 문장 봉인
> `compareAndSet(기대, 새값)`은 DB의 `UPDATE WHERE` 낙관 CAS를 메모리 연산 하나로 압축한 것 — **루프(읽기→판정→CAS)**로 감싸면 틈이 사라지는 게 아니라 **무해해진다**(낡은 정보로는 절대 못 쓴다). 단 CAS는 변수 하나만 묶으므로 **2필드 상태는 불변 객체+AtomicReference**로 하나로 만들어 교체하고 — 그럼에도 우리 버킷이 synchronized로 남는 이유는 **경합이 낮으면 락 비용이 ~0이라 결정 축이 단순성으로 넘어가기** 때문이다(DB 비관 락과 가격표가 다르다).

## 6. 실습 결과 — Bucket4j 적용 (experiment/bucket4j 브랜치, 커밋 4767d16c)
- 직접 구현을 **공개 API 그대로** 둔 채 어댑터로 교체: 70줄 로직 → 설정+위임 (본문 -70/+49줄, 그중 절반이 검증·javadoc). **인터셉터·설정·테스트 변경 0줄.**
- 매핑: `tryConsume()` → `bucket.tryConsume(1)` / `tryConsume(maxWait)` → `asBlocking().tryConsume(1, nanos, strategy)` (**fail-early 내장** — 마감 내 못 차면 즉시 false) / `retryAfterSeconds()` → `estimateAbilityToConsume(1).getNanosToWaitForRefill()` 올림.
- 결정적 테스트 호환: 가짜 시계 → `TimeMeter` 주입, 가짜 sleeper → `BlockingStrategy.park` 주입 — **기존 테스트 전부 무수정 그린**(전체 스위트 포함). 우리가 만든 "시계·잠 주입" 설계가 Bucket4j의 확장점과 정확히 같은 자리였다는 뜻.
- 한 가지 차이(내부 semantics): Bucket4j 블로킹은 재확인 루프가 아니라 **예약(reserve) 후 잠들기** — 토큰을 미리 차감하고 그만큼 자니 herd 재경합 자체가 없다. 우리 구현(깨어나 재확인)과 결과는 같고 전략이 다름 — 씨앗으로만.
- **본선 반영 결정(같은 날 추가, 커밋 247857f0)**: 학습자 발의로 feat 브랜치에도 교체 반영. §4 판단(성능·단순성 축에선 바꿀 이유 없음)을 뒤집은 게 아니라 **다른 축을 얹은 것** — "한 번도 안 써본 도구를 써보는 경험 자체가 학습 목표, 매 프로젝트마다 직접 구현할 수는 없다." 직접 구현은 git 이력(685a4a6e)과 이 로그에 보존.
- 부수 학습(반영 중 질문): `REFILL_WINDOW_SECONDS`는 왜 설정으로 안 빼나 → **손잡이가 아니라 단위 환산 상수**(double rate→정수 토큰/기간; 값을 바꿔도 행동 불변). 설정 외부화 기준 = *"운영자가 바꾸면 의도된 행동 변화가 생기는 값(정책)만"* — 행동이 안 변하는 값을 빼면 가짜 손잡이(혼란+오조작 표면만 증가).
- 부수 학습 2(코드 리뷰 질문): "왜 `BlockingStrategy`를 생성자 주입하지 않고 `BackoffSleeper`를 받아 안에서 람다로 감쌌나" → `BlockingStrategy`는 **남의 집 타입**(io.github.bucket4j) — 생성자에 올리면 모든 조립처(TossConfig 등)가 라이브러리를 import하고 교체 파급이 전체로 번짐. **남의 타입은 어댑터 안에 가둔다** — 지금 Bucket4j를 아는 파일은 정확히 1개. log_38 ACL(토스 DTO 가두기)의 동형 재등장(학습자가 인출로 도달: "임포트 최소화 + 나중에 자는 방식 교체 쉬움").
