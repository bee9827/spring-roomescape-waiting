# 57. 락 발자국 스윕 — 집합 제약은 UNIQUE로, 카운트 제약은 실존 행 앵커로, 스테일은 안전한 방향만

**날짜**: 2026-07-06 (log_56과 같은 날 — arc '락 발자국 스윕' 한 칸, **당일 닫힘**)
**학습 범위**: log_56 리뷰가 실증한 잔존 카테고리(JOIN FOR UPDATE 4곳 + 승격 gap-then-insert)를 한 패스 스윕. 새 개념 학습이 아니라 **전날 세운 원리를 5개 현장에 대입하는 적용 사이클** — 현장마다 불변식이 달라 처방이 갈리는 것이 학습 내용. 재방문: log_56(gap·발자국·UNIQUE 백스톱), log_29(FOR UPDATE 스냅샷 우회 — 동작 레벨 사용), log_25(대기 5명 규칙).

분류: DB·트랜잭션(동시성) — 코드 적용 + 실전 판단

## 1. 제약의 종류가 장치를 정한다 (이 arc의 축)
- **집합 제약**(중복 금지): DB가 원자적으로 봐줌 → UNIQUE로 내린다 (log_56).
- **카운트 제약**(개수 ≤ N): 여러 행을 "세는" 관찰이 필요 = 본질적 check-then-act. `CHECK`는 행 안만 보고, 순번 컬럼+UNIQUE 인코딩은 가능하나 취소 시 재부여 등 "카운트 문제를 순번 관리 문제로 바꿀 뿐"(학습자 도출) → **직렬화 장치(락)가 정당하다.**
- 락이 성립하는 조건은 log_56의 역: **잠글 실존 행이 있는가.** 대기는 예약이 있어야만 가능(`orElseThrow`) → 앵커는 항상 실존 행 = record 락 = 진짜 한 줄 세우기. 빈 슬롯(gap)과 결정적으로 다른 지점.

## 2. 현장별 판정 (전부 학습자 도출)
| 현장 | 불변식 | 처방 |
|---|---|---|
| 대기 신청 앵커(1a) | 최대 5명(카운트) | **락 유지**, id만 잠금(JOIN 제거) + 무잠금 findById 로드 |
| 큐 읽기(1b) | — (앵커가 이미 직렬화) | **FOR UPDATE 제거** — 무잠금 스냅샷 |
| 승격 사전검사(2a) | 슬롯 비었으면 승격(멱등) | **FOR UPDATE 제거**(gap 안티패턴) — 진실은 UNIQUE 백스톱 |
| 첫 대기자(2b) | 이중 승격·승격↔취소 금지 | **락 유지**(실존 행), JOIN 제거 |
- **무잠금 SELECT는 JOIN이 백 개라도 락 0** — "FOR UPDATE는 읽은 행 전부"의 대우. 1b에서 걷어낼 것은 JOIN이 아니라 잠금이었다.
- **스테일의 방향성**: 1b 스냅샷이 낡으면 "4명인데 5명인 줄 알고 거절"만 가능(늘리는 쪽은 앵커에 줄 서 있으니 역방향 불가). **안전한 쪽으로만 틀리는 낡음은 수용** — log_46 낙관 가드와 같은 결.

## 3. DuplicateKey의 의미론 — 예외 하나에 시나리오 둘
- 승격 INSERT가 UNIQUE에 막히는 원인: ① 다른 워커가 먼저 승격(그 트랜잭션이 waiting도 지웠음) ② **어드민 직접 예약**(createByAdmin은 대기열 검사 없음)이 슬롯 선점 — 이 경우 첫 대기자는 승격 안 됨.
- 처방(학습자): catch → **waiting은 안 건드리고 태스크만 소비(markDone)**. "다음 취소가 새 태스크를 적재하니 잃는 것이 없다" — 지우는 코드가 실제로 뭔가를 지우는 유일한 경우가 ②(억울한 증발)라서, 모든 경우에 "안 지운다"가 같거나 낫다.
- **전제 명시 사건**: 학습자의 "지우면 돼"의 목적어(promotion 태스크)를 코치가 waiting 행으로 오독하고 허수아비 반박 → 학습자 정정. log_56 "바꿀 것"(전제 명시)이 하루 만에 두 번째 실증.

## 4. 패자 워커의 여정 — 스냅샷 우회의 실전 사용
- B가 A의 첫 대기자 락에서 대기 → A 커밋(승격+삭제) → 깨어난 B의 잠금 읽기는 **스냅샷이 아니라 최신 커밋을 읽어**(current read, log_29 동작 레벨) 다음 대기자를 봄 → 승격 시도 → UNIQUE가 막음 → catch → 소비. 어느 갈래로도 대기자 증발·이중 승격 없음.
- 리뷰 검증(세밀): B의 read view는 락 대기 **전** 일반 SELECT(existsBySlotKey)에서 고정 — 잠금 읽기는 read view를 안 쓰고 안 만든다. findById(옛 스냅샷)가 어긋나는 유일한 경우 = 락 걸린 행이 스냅샷 이후 삽입 → empty → skip → 안전(새 태스크가 승격 보장). **==5 불변식의 린치핀** = 앵커 락 "이후" 첫 일반 SELECT가 read view를 만든다는 순서 — 주석으로 봉인.

## 5. 리뷰 발견 — "잠기는 범위는 실행 계획이 정한다" (log_56 씨앗 즉시 회수 2호)
- `ORDER BY id LIMIT 1 FOR UPDATE`: 슬롯 인덱스는 member_id 순 정렬이라 옵티마이저가 "PK를 id순 스캔" 플랜을 고르면 **다른 슬롯 행까지 잠금** — 스윕이 없애려던 바로 그 부류가 플랜 변덕으로 재발 가능. 처방: 정렬 유인 제거 — 슬롯 행 전부(≤5, 도메인 규칙) 잠그고 **min(id)는 Java에서**.
- 남은 관찰(리뷰 LOW): 생성 가드 waitings gap ↔ 대기신청 앵커 사이 좁은 데드락 창(경합+스테일 스냅샷 중첩 시, 이전에도 존재) — 씨앗으로만.

## 한 문장 봉인
> 락을 지울지 남길지는 이념이 아니라 **불변식의 종류**가 정한다 — 집합 제약(중복)은 UNIQUE로 내리고, 카운트 제약(≤N)은 내릴 곳이 없으니 **실존 행 앵커의 record 락**으로 임계구역을 만든다(빈 슬롯 gap과 달리 진짜 직렬화됨). 잠금 읽기는 최소 발자국(JOIN 금지·id만)으로, 무잠금 읽기는 JOIN이어도 공짜로, 스테일은 **안전한 방향으로만 틀리면 수용** — 그리고 잠기는 범위는 SQL이 아니라 **실행 계획**이 정하므로 플랜까지 고정해야 발자국 설계가 끝난다.
