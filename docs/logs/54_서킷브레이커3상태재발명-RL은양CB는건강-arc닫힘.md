# 54. 서킷 브레이커 3상태 재발명 — Rate Limit은 양, 브레이커는 건강 (arc 'step3 Rate Limit' 닫힘)

**날짜**: 2026-07-03 (같은 날 세 번째 사이클 — 52 구현, 53 인출 폐쇄, 54 arc 마감)
**학습 범위**: hot arc 'step3 Rate Limit'의 #5 칸(마지막) — 흐름 파악. 재방문: log_49(이름만 스침), log_50(§5 씨앗 "병목이 외부 한도면 우리가 못 올린다"), log_52(bounded wait·50ms), log_53(read timeout 5초·스펙트럼).

분류: 외부 결제 연동 — 흐름 파악 (arc 마감 칸)

## 1. Rate Limit의 사각 — 죽은 토스 시나리오
- 토스 전면 장애 10분. 토큰은 *우리 시계*로 차니(52) 호출은 **초당 20개씩 계속 출발**하고, 각각 read timeout **5초**(53)까지 매달렸다 실패한다.
- 곱하면(학습자 계산): 20/s × 5s = **동시 ~100 스레드**가 시체에 매달림 — 톰캣 기본 풀(~200)의 절반. 토스와 무관한 요청(예약 조회)까지 느려진다 — **남의 장애가 내 장애로 전염**.
- 진단(학습자): *"문제는 죽은 서버 때문에 우리 자원이 계속 낭비되는 것."* Rate Limit이 재는 건 **양**(호출이 너무 많은가)인데 지금 문제는 **건강**(호출이 성공하기나 하는가) — 한도는 완벽히 지켜지는 중이다. *재앙을 초당 20개씩 정량 배급하면서.*

## 2. 서킷 브레이커 — 3상태를 전부 학습자가 도출
- 시스템은 이미 정보를 쥐고 있다(지난 10분 전부 실패) — 그걸 쓰면: **열림(open)**. 다음 호출을 토스 근처에도 안 보내고 **0ms 즉시 실패**. 우리 스레드 해방 + 죽어가는 상대도 두들겨 맞지 않고 숨통(양쪽 회복).
- **딜레마**: 상태를 아는 유일한 수단(호출)을 우리 손으로 끊었다 — 살아난 걸 어떻게 아나? 학습자 답 둘 다 실전에 존재:
  - **정찰병** = **반열림(half-open)** — 차단 후 일정 시간 지나면 소수 호출만 통과. 성공→닫힘(전면 재개), 실패→다시 열림+타이머 리셋.
  - 운영자 **수동 전환**(force open/close) — 실전 라이브러리에 있으나 최후 수단(장애는 새벽 3시에 온다).
- **닫힘→열림 트립 조건**: "단위시간당 timeout 개수"에서 출발 → 개수의 함정을 시나리오 대비로 자가 수정(*낮 10/1,000 vs 새벽 10/10 — 개수는 같은데 판단이 다르다 → "표본이 적은 건 생각 못 했네"*) → **실패율 AND 최소 표본 수**. 실패는 timeout뿐 아니라 즉시 5xx도 포함. 비율 임계는 100%가 아니라 다이얼(보통 ~50% — 절반이 죽는 서버도 이미 병자).

## 3. 재발명 대조 — Resilience4j 설정과 1:1
| 학습자가 도출한 것 | Resilience4j 설정 |
|---|---|
| 실패 비율 기준 | `failureRateThreshold` (기본 50%) |
| 최소 표본 수 | `minimumNumberOfCalls` |
| 단위 시간(집계 창) | `slidingWindow` |
| 차단 후 정찰까지 대기 | `waitDurationInOpenState` |
| 정찰병 수 | `permittedNumberOfCallsInHalfOpenState` |

상태 3개(closed/open/half-open)부터 트립 조건까지 — 개념을 통째로 재발명한 뒤 라이브러리에 대조.

## 4. 상호보완 — 왜 둘 다인가
- 봉인(학습자 문장): **"Rate Limit은 요청이 너무 많을 경우 나가는 양을 조절하고, 서킷 브레이커는 외부 API에 장애가 났을 경우 우리 서버의 스레드를 지킨다."**
- 풀면: **RL = 평시**(상대가 건강할 때) 나가는 양이 quota(상대와의 약속, log_50)를 넘지 않게 — 429 예방. **CB = 장애 시**(상대가 아플 때) 건강을 감지해 호출 자체를 끊어 우리 자원 보호. **재는 축이 달라(양 vs 건강) 서로를 대신 못 한다** — 그래서 둘 다.
- 교정 1건: "RL이 나가는 *스레드* 양을 통제" → 통제 대상은 스레드가 아니라 **호출량**. 스레드를 지키는 건 CB(와 bounded wait의 마감, log_52)의 일 — 죽은 토스 시나리오가 그 증거(RL 완벽 작동 중에 스레드 100개 잠식).

## arc 'step3 Rate Limit' 닫힘 — 다섯 칸 한 줄씩
1. **#1·#2 (log_50)**: capacity=순간 버스트 / refill=지속 평균. 인바운드(내 용량) ≠ 아웃바운드(상대 몫) — 비교규칙 없음, 워커·재시도가 인바운드를 우회.
2. **#3 (log_52)**: 거부 정책 = 대기시간 × 동시대기자수 → 아웃바운드만 bounded wait로 전환 구현. synchronized/check-then-act 첫 대면.
3. **#4 (log_53)**: 재시도 안전성 = "토스가 어디까지 아나" 스펙트럼 — 인출만으로 닫힘.
4. **#5 (log_54)**: RL의 사각(건강) → 서킷 브레이커 3상태 재발명.
- 씨앗(본문에만): **서킷 브레이커 코드 적용** — 이 칸은 흐름 파악까지. 구현하게 되면 직전 arc의 '재시도 정책(워커 재시도 바운드·언제 포기→보상)' 칸과 한 묶음이 자연스럽다(트립·백오프·포기가 한 가족).

## 한 문장 봉인
> Rate Limit과 서킷 브레이커는 **재는 축이 다르다** — RL은 *양*을 재서 **평시**에 상대와의 약속(quota)을 지키고, CB는 *건강*(실패율 AND 최소 표본)을 재서 **장애 시** 우리 스레드를 지킨다. 죽은 서버 앞에서 RL은 재앙을 정량 배급할 뿐이므로(20/s × 5s = 동시 100 스레드), 건강 축은 **closed→open(즉시 실패)→half-open(정찰)** 의 별도 장치가 맡는다 — 하나로 다른 하나를 대신할 수 없어 둘 다 건다.

## 학습법 회고

### 잘된 것
- **칸 전체가 시나리오 하나에서 출발해 학습자 도출로 끝났다.** "죽은 토스" 문제 제시 → 100 스레드 계산 → 양vs건강 진단 → open 상태 → 정보 딜레마 → half-open → 트립 조건까지, 새 개념(서킷 브레이커)을 학습자가 통째로 재발명. 마지막 Resilience4j 1:1 대조가 "내가 만든 게 진짜였다"는 보상으로 작동.
- **개수→비율 자가 수정** — 낮/새벽 대비 시나리오 하나로 "표본이 적은 건 생각 못 했네"를 스스로 말함. log_50의 "상황 뒤집기로 규칙 폐기"와 같은 도구가 안정적으로 작동.
