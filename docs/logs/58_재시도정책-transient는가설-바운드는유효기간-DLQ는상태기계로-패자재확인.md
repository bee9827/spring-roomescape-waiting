# 58. 재시도 정책 — transient는 가설, 바운드는 그 유효기간 + DLQ를 상태 기계로 + CAS 패자는 추측 말고 재확인

**날짜**: 2026-07-07 (arc '재시도 정책' 1칸 — 칸 닫힘, arc는 가족 씨앗 2칸 남음)
**학습 범위**: hot 칸 "워커 재시도 횟수·백오프·언제 포기→보상"(예고 log_46 §1·48 §5·#36). 4개 워커의 무한 "다음 주기 재시도"에 바운드와 결말을 설계·구현 — 하다가 리뷰가 벌린 **confirm의 시간 경계 구멍**(돈 고아 문제)까지 닫음. 재방문: log_44(root-cause 분류), log_46(조회로 진실), log_47(상태 CAS), log_48(NEEDS_REFUND 경첩), log_35(인증 vs 승인), log_56(데드락 패자).

분류: 외부 결제 연동 + DB·트랜잭션 — 흐름 파악 + 코드 적용 + 실전 판단

## 1. 출발 인출 — "포기가 없으면 실패라는 결말이 존재하지 않는다"
- 현재 4개 워커 전부 `catch → log.warn → 다음 주기`(무한). 시도 횟수를 기억하는 곳 없음.
- 학습자 시나리오: 승격 태스크가 영원히 실패하면 — 새치기 가드는 직접 예약을 막고, 새 태스크는 취소가 있어야 적재되는데 취소할 예약이 없음 → **슬롯 영구 림보**. 로그가 쌓이는 문제가 아니라 비즈니스가 멈추는 결함.
- 기존 경첩 재방문: confirm의 `BusinessRule/EntityNotFound → 즉시 NEEDS_REFUND`(log_48)는 **재시도 없는 포기**였다 — 왜? 지워진 예약은 백 번 재시도해도 안 돌아온다.

## 2. 실패의 축 — transient(지나가는) vs permanent(머무는)
- **transient**: 원인이 지나가는 상태(네트워크 순단·상대 다운·락 경합·**데드락 패자**[log_56 처방 연결]) → 재시도가 새 기회를 얻음. / **permanent**: 세상이 확정됨(엔티티 소멸·규칙 위반) → 같은 답만 반복.
- 어원 연결: 라틴어 transire(지나가다) — Java `transient` 키워드(직렬화 제외 = 영속하지 않는 값)와 같은 뿌리.
- **바운드의 정체(오늘의 축)**: transient 분류는 **가설**이다. N회 연속 실패 = 가설 기각 → permanent로 **재분류**. 바운드는 처벌이 아니라 가설의 유효기간. (학습자 1차 답은 "재시도의 전제=멱등성" — 전제와 *만료*를 구분하는 정정을 거침.)

## 3. 포기 ≠ 버리기 — DLQ를 상태 기계로 번역
- 학습자 막힘: "NEEDS_CHECK도 Refund도 그냥 버릴 순 없어" → 등식 하나 깨니 풀림: 포기 = 삭제가 아니라 **격리 보존 + 알림 + 수동 처리**(학습자 도출).
- 업계 이름: **DLQ(dead letter queue)** / poison message. 우리는 메시지 큐가 아니라 DB 상태 기계 세계 → 격리 테이블(이사)이 아니라 **상태 전이**: `DEAD` 상태로 바꾸면 폴링 WHERE에서 자연 이탈 + 행 보존(FK·이력·`UNIQUE(reservation_id)` 백스톱 무사 — DEAD 주문이 새 결제를 막는 것도 올바른 행동, 테스트로 봉인).
- ⓐ(출신별 상태 CHECK_DEAD/REFUND_DEAD) vs ⓑ(단일 DEAD+이전상태 컬럼): ⓑ는 "status 한 칸=진실 전부" 원칙이 깨지고 컬럼 간 불변식(규율) 발생 / ⓐ의 enum 증식 우려는 **수렴 중 상태에만 짝이 생기므로**(종착 상태는 짝 없음 — `deadCounterpart()`로 코드화) 선형 억제 → ⓐ. 상태 증식의 실전 감각: 토스 Payment도 8상태 — 늘어나는 축은 생애주기지 DEAD가 아님.
- 잔여물 축 둘(학습자 자기모순 정정): reconcile·refund = **돈** / promotion·abandonment = **슬롯과 줄**("나머지는 자동으로 돌아간다"는 자신의 오프닝 시나리오와 모순).
- **DuplicateKey의 의미론**: 승격 INSERT가 UNIQUE에 막힘 = ①다른 워커가 승격(waiting도 지워짐) or ②어드민 직접 예약(대기자 승격 안 됨). "지운다"의 목적어 오독 사건(코치가 waiting으로 읽음 → 학습자 정정: **태스크만 소비, waiting은 줄 유지** — 다음 취소가 새 태스크를 적재하니 잃는 것 없음).

## 4. 구현 디테일 (커밋 096b3075) — 카운터의 불변식
- `attempt_count`는 "**현재 상태에서의** 실패 수"만 센다 → 리셋은 상태 전이와 **같은 UPDATE 문** 안에(원자성 — 쪼개면 "전이 없이 리셋만" 같은 깨진 조합 가능 / 불변식은 그 쌍을 쓰는 자리에 — log_41 "매 경로 기억은 취약"의 동형). 증가도 SQL로(`+1`) — Java에서 읽고 더하면 check-then-act(log_52 AtomicLong 동형).
- 리뷰가 잡은 카운터 구멍 3: **M2** 같은 상태 재기록(recheck NEEDS_CHECK→NEEDS_CHECK)이 카운터 리셋 → 사용자가 바운드를 무한 우회 → CASE WHEN으로 "바뀔 때만" 리셋. **M3** 증가에 상태 가드 없음 → 전이 직후 낡은 실패가 새 상태에 유령 계상(한도가 1 일찍) → `AND status=:expected`. **L1** 실패 기록 자체의 실패(원인이 DB 장애면 기록도 죽음)가 배치를 중단 → 기록 메서드 자체 격리.

## 5. confirm의 시간 경계 — 학습자 반격이 처방을 셋으로 갈랐다
- 리뷰 H1: CAS 패자가 무조건 CONFIRMED 반환("누군가 CONFIRMED로 갔겠지" **추측**). 학습자 반격: "애초에 사용자가 개입 못 하는 선에서 DEAD돼야 하는 것 아냐? TTL이 강제하잖아" → 검증: **맞는 부분**(EXPIRE_DEAD는 TTL+5주기 뒤라 결제창이 이미 죽음 — 리뷰 시나리오는 과장) + **균열 둘**(①그 보장이 기계가 아니라 설정 관습인데 기본값 TTL 30 = 창 30으로 딱 붙음 ②창 마감 직전 승인은 승인(토스)↔기록(DB) 사이 **δ초**가 항상 존재해 시간으로 못 닫음).
- 시나리오 두 개로 분해(학습자 질문 "시나리오가 어떻게 되는 거지?"): **A 경계 경합**(29:59:58 승인 → 30:00:00 expire FAILED → 30:00:01 CAS 패배 → "CONFIRMED" 거짓말, 돈 고아) / **B 뒤늦은 복귀**(인증만 하고 이탈 → 만료 후 돌아옴 → 죽은 주문에 돈이 나갔다 환불되는 왕복 + FAILED 부활).
- 처방 셋, 각자 다른 구멍(커밋 e09cb83f): ①**TTL 35분**(시간 분리 — A의 빈도 제거) ②**패자 재확인**(기계 — 지면 실제 상태를 다시 읽고[log_46 조회로 진실], CONFIRMED면 참말, 아니면 NEEDS_REFUND 전이[log_48 경첩]로 고아 돈을 환불 시야에) ③**진입 가드**(B — `validateConfirmable`: 죽은/실패 주문은 **돈이 움직이기 전에** 거절, 학습자: "아예 돈이 안 나가게 만들면 문제가 발생하지 않잖아") + 확정 주문 재호출은 승인 없이 멱등 CONFIRMED.
- 테스트: FakePaymentGateway에 **승인 직후 훅**(δ에 끼어드는 경합 흉내)·호출 카운터("돈이 아예 안 나감" 증명) — 경합을 결정적으로 재현. 전체 300 그린.

## 한 문장 봉인
> "다음 주기 재시도"만 있는 시스템엔 실패라는 결말이 없다 — **transient는 가설이고 재시도 바운드는 그 가설의 유효기간**이다. 기각되면 permanent로 재분류해 **DEAD 상태로 격리**(DLQ의 상태 기계 번역: 폴링에서만 빠지고 행은 보존, 격리는 조용하면 안 됨)하고, 카운터의 불변식("현재 상태의 실패 수")은 **상태를 쓰는 그 UPDATE 문 안에** 산다. 그리고 CAS 패자는 결과를 **추측하지 말고 다시 읽어라** — 승인(외부)과 기록(DB) 사이의 δ초는 시간 분리로 좁힐 순 있어도 없앨 순 없으니, 진실 재확인과 진입 가드(돈이 움직이기 전 거절)가 기계로 닫는다.

## 씨앗 (본문에만)
- 지수 백오프 — 지금은 폴링 주기(60s)가 고정 백오프 역할. 건별 지수 증가(attempt 기반 다음 시도 시각)는 미구현.
- DEAD의 운영 도구 — 격리 건 조회·수동 재개(DEAD→NEEDS_*) 어드민 API/알림 채널. 지금의 "사람 호출"은 ERROR 로그뿐.
- 고아 PENDING 예약(승격 유래, 주문 없음) 경로는 카운터 없음 — DB-로컬 실패라 보류(주석).
- resolveLostConfirm의 이중 패배(희귀) 처리 — 마지막 재확인 한 번으로 봉합했는데, 수렴 루프의 일반형(재시도 몇 번?)은 미탐구.
