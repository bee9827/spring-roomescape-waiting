# 53. 재시도 안전성 = "토스가 어디까지 아나" 스펙트럼 + 429는 발화자가 둘 + 멱등키의 진짜 가치

**날짜**: 2026-07-03 (log_52와 같은 날, 두 번째 사이클)
**학습 범위**: hot arc 'step3 Rate Limit'의 #4 칸(read timeout 재시도 vs 429 재시도 — 멱등성) — **새 설명 0, 인출만으로 닫힘.** 학습자의 "#4 닫힌 거 아니야?"라는 감이 정확했고, 조각들(log_44 멱등 두 겹, log_36 멱등키, log_46 워커 수렴)을 하나의 스펙트럼으로 묶는 것 + 오늘 생긴 모서리(log_52의 아웃바운드 거부) 얹기만 남아 있었다.

분류: 외부 결제 연동 — 흐름 파악→실전 판단 (재방문: log_36·44·46·50·52)

## 1. 재시도 안전성 스펙트럼 — "나가는 토스 호출 한 건의 생애" 기준
| 상태 | 토스가 아는 정도 | "안 됐다" 확신의 출처 | 재시도 요구 |
|---|---|---|---|
| **아웃바운드 거부** (log_52 bounded wait 폴백) | 존재도 모름 — 우리 문턱도 안 넘음 | **우리 자신** (안 보냈으니까) | 없음 — 재시도가 그냥 첫 시도 |
| **토스 429** | 도달했고 **알지만 처리 안 함** — "토스 내부 인바운드에 걸린 것"(우리 인바운드 리미터와 대칭) | **토스의 응답** (429 자체가 증거) | 그대로 재전송 안전 |
| **read timeout** | 도달·처리 여부 **모름** (됐을 수도) | 없음 — 그래서 위험 | **멱등키** + NEEDS_CHECK로 워커 수렴(조회로 진실, log_46) |
| **성공 200** | 알고 처리함 | — | 불필요 |
- 아래로 갈수록 토스가 아는 게 많아지고, 재시도가 요구하는 안전장치가 무거워진다. 429보다 아웃바운드 거부가 *더* 안전한 등급(429는 왕복이라도 했다).

## 2. 멱등키의 진짜 가치 — 재시도를 특수 케이스가 아니게
- 멱등키 없이도 중복은 어찌어찌 막힌다 — orderId 중복으로 토스가 **에러**를 주니까. 하지만 그러면 정상 로직 한복판에 "이 에러가 사실은 성공인가?"를 판별하는 **분기가 오염**된다(학습자 자가 도출).
- 멱등키가 있으면: 재시도가 첫 시도와 **같은 성공 응답을 재생**받아 재시도 경로와 정상 경로가 하나로 합쳐진다. log_36의 "exactly-once **효과**"의 실체 — 에러를 주는 방식이었다면 그 이름이 성립하지 않았다.
- 즉 멱등키의 가치는 *중복 방지 그 자체*가 아니라 **재시도를 특수 케이스가 아니게 만드는 것**.

## 3. 429는 발화자가 둘 — 프레임 없이 말하면 혼선
- **우리 → 손님** 429: 인바운드 리미터가 컨트롤러 앞에서 거절(+Retry-After). 이 손님 요청은 토스 호출을 *만들지도 않음* → 위 스펙트럼 **밖**(재시도도 손님 몫, 워커 몫 아님).
- **토스 → 우리** 429: 우리 아웃바운드가 토스 quota 초과. 스펙트럼 안의 429는 이것뿐.
- 실전 감각(학습자): 토스 429가 자주 보이면 우리 아웃바운드 상한이 quota 대비 과대 — 조절하거나 **분리**해야. PG사는 메서드별 quota가 다르다(쓰기≠읽기 — 승인은 쓰기, 워커의 reconciliation은 *조회*라 다른 한도를 탈 수 있음).

## 4. 오류 교정 기록
- **방향 용어 혼동 2회째**: "우리 거부(=아웃바운드 버킷)"를 "인바운드로 막는다"로 표현 — log_50에서 교정했던 지점의 재발. 좌표 인출(WebMvcConfig의 손님용 vs TossConfig의 토스용)로 재교정. → 관찰 항목: 인/아웃바운드 방향 단어가 아직 자동화 안 됨.

## 한 문장 봉인
> 재시도 안전성은 **"토스가 어디까지 아나"**로 등급이 매겨진다 — **아웃바운드 거부**(존재도 모름, 확신=우리)와 **토스 429**(받았지만 미처리, 확신=토스 응답)는 그대로 재전송해도 안전하고, **read timeout**(모름)만이 **멱등키**(재시도를 특수 케이스가 아니게)와 **워커 수렴**(NEEDS_CHECK→조회로 진실)을 요구한다. 단, "429"는 발화자가 둘(우리→손님 / 토스→우리)이라 프레임 없이 말하면 다른 얘기가 된다.

### 잘된 것
- **칸 하나가 새 설명 0으로 닫혔다.** 학습자의 "닫힌 거 아니야?" 감지 → 인출 검증으로 확인 — 재방문 판정을 학습자가 먼저 한 첫 사례. 백로그 칸이 "남은 학습"이 아니라 "묶기만 남은 것"일 수 있음을 보여줌.
