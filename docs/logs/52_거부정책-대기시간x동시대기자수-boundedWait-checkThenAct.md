# 52. 거부 정책 = 대기시간 × 동시대기자수 — 아웃바운드를 bounded wait로 + synchronized 첫 대면

**날짜**: 2026-07-03
**학습 범위**: hot arc 'step3 Rate Limit'의 #3 칸(fail-fast 거부 vs 블로킹 대기) — 개념 판단 → 코드 적용 → 구현 리뷰(동시성)까지 한 사이클. 재방문: log_50(#1·#2 버킷·독립 한도), log_44(read timeout — "기다림에는 마감" 원리 재사용), log_49(RetryAfterInterceptor·BackoffSleeper), log_47(CAS — §4 씨앗으로 연결).

분류: 외부 결제 연동 / 동시성 — 실전 판단 + 코드 적용

## 1. 거부 정책 두 갈래와 결정 공식
- **B fail-fast(거절)**: 429 + `Retry-After`로 즉시 돌려보냄. Retry-After는 **B 전용** — 거절 응답에 얹는 헤더라, 붙잡아 성공시키는 A엔 얹을 429 자체가 없다.
- **A blocking(블로킹 대기)**: 토큰이 찰 때까지 붙잡고 기다림.
- 결정 손잡이 둘(학습자 자가 도출): ① **대기 시간** — 짧으면 거절→손님 재시도 *왕복 비용*이 잠깐 붙잡기보다 비쌈(A 유리), 길면 스레드가 못 버팀(B 유리). ② **동시 대기자 수** — 유계냐 무계냐. 합치면 **블로킹의 위험 = 대기 시간 × 동시 대기자 수**. 한 축만으론 판단 불가(50ms라도 무계 손님 5,000이 동시에 잠들면 톰캣 풀 ~200은 그 전에 바닥).
- 적용: **인바운드 = B 유지**(무계 손님 + 스레드 풀 보호 + Retry-After 보고 재방문 가능). **아웃바운드 = A로 전환**(호출자가 내부 워커+결제 스레드로 유계, refill 20/s → 대기 ~50ms로 짧음, 거절하면 워커가 재호출해 토스 quota를 되레 낭비). → "둘 다 거절"이 아니라 **방향마다 답이 갈리는 게 정상**.
- B의 약점: 부담이 전부 손님에게 넘어가고, 다들 비슷한 타이밍에 되돌아와 몰림(**thundering herd**). A는 서버가 붙잡아 눌러줌(**backpressure**).

## 2. 순수 A가 아니라 bounded wait 하이브리드
- 무한 대기 금지 — A를 고른 전제("대기가 짧다")는 설정 실수·예상 밖 폭주로 깨질 수 있고, 그때 스레드가 하염없이 붙잡힘. **기다림에는 언제나 마감**(read timeout과 동일 원리, log_44). 마감이 터지면 **B로 폴백**: 예외 → 기존 오류 경로 → 워커가 다음 사이클에 재시도. 즉 *"짧으면 기다려 성공시키고(왕복 절약), 길어지면 끊어 스레드를 살린다"* — 실무 라이브러리(Guava RateLimiter, Resilience4j)도 이 모양.
- 이 블로킹은 **토스를 기다리는 게 아니다** — 우리 버킷이 우리 시계로 차는 걸 기다리는 순수 우리 쪽 산수. `[토큰 대기]→[토스 호출]→[응답/오류 처리]` 중 첫 칸만 변경, 토스 오류(429/5xx/timeout)는 기존 장치(백오프·read timeout·워커) 그대로.
- 명명은 j.u.c. 관례: 즉시 실패 `tryAcquire()` / 무한 대기 `acquire()` / **시간 상한 대기 `tryAcquire(timeout)`** — "기다리긴 하지만 실패할 수 있으니 여전히 try". → `tryConsume(Duration maxWait)` 오버로드로 채택.
- **fail-early**: 마감 안에 못 찰 걸 *계산으로 미리 알면* 1ms도 안 기다리고 즉시 거절 — 결과가 같은 실패라면 어차피 실패할 대기에 스레드를 붙잡아두지 않는 것(효율)이 이긴다(퀴즈 Q2 인출).

## 3. 구현 좌표 (spring-roomescape-waiting, step3)
- `TokenBucketRateLimiter.tryConsume(Duration maxWait)` **오버로드 추가** — 새 메서드 추가는 기존 호출자(인바운드의 `tryConsume()`) 무영향. 인바운드/아웃바운드가 공유하는 건 *클래스뿐, 버킷 인스턴스는 별개* — 한 클래스가 두 정책을 제공하고 호출자가 고르는 구조.
- `BackoffSleeper` 이사: `payment.toss` → `common.ratelimit`. 리미터가 "자는 방법"이 필요한데 common이 payment를 import하면 **의존 방향 역행**(log_40) → 쓰는 쪽을 내리는 게 아니라 *공용 물건을 위로*.
- `OutboundRateLimitInterceptor`: `tryConsume(maxWait)` false → 기존 `OutboundRateLimitException`(오류 경로 재사용). `max-wait: 2s` 설정 외부화.
- `retryAfterSeconds()` 재사용 안 한 이유: 초 올림은 **Retry-After 헤더용 표현**(HTTP 스펙이 초 단위 정수) — refill 20/s면 50ms 필요를 1초로 **20배 과잉 수면**. → 학습자 제안 리팩터링: `private nanosUntilNextToken()`에 공식을 한 곳만 두고, 초 버전은 그걸 올림해 *표현*만 담당. "언제 차는가의 진실은 한 군데, 초/나노는 각자의 표현."
- 테스트: 가짜 sleeper가 **가짜 시계를 잔 만큼 전진** → 실제로 안 자고 "기다렸다 성공"(잔 시간 500ms까지)·"마감 불가 시 즉시 거절(잠 0회)"을 결정적으로 검증.

## 4. synchronized 첫 대면 — check-then-act 틈은 원자 연산으로 못 닫는다
- **race condition / check-then-act**: `①if(tokens>=1) ②tokens-=1`은 두 줄이고 CPU는 아무 데서나 스레드를 갈아끼움. tokens=1에 A·B 동시 도착 → 둘 다 ①에서 1을 보고 통과 → -1, "capacity개만 통과" 약속 파괴.
- **AtomicLong 후보 탈락**(학습자 제시 → 탈락법): 원자성은 *연산 하나하나*(`decrementAndGet` 내부)만 보장. 깨진 곳은 ①*안*이 아니라 **①과 ② 사이** — 그 틈은 그대로 남는다. → **묶음 전체를 문 하나로**: `synchronized` = "이 구역엔 한 번에 한 스레드만".
- **잠은 문 밖에서**: 락을 쥔 채 자면, 토큰이 있어 바로 통과할 스레드까지 전원 문 앞에 블락 — "잠깐 기다리는 정책"이 "모두를 강제로 기다리게 하는 정책"으로 변질.
- 그 **대가**: 자는 동안 문이 열려 있어 깨어나 보면 토큰을 다른 스레드가 채갔을 수 있음 → `if`가 아니라 `while` 재확인. 여럿이 잠들었다 비슷한 타이밍에 깨어 몇 개 안 되는 토큰에 몰리는 것 = **thundering herd 미니어처**(서버 사이 현상이 스레드 사이에서 동형 재현).
- 씨앗(본문에만): `compareAndSet` — 확인-행동을 진짜 한 방에 묶는 **lock-free** 방식. log_47의 낙관 CAS(`UPDATE WHERE status=expected`)와 같은 뼈대가 메모리 레벨에도 있다. 재방문거리.

## 5. 오류 교정 기록
- 산수: 토큰 하나 차는 시간을 20ms로 인출 → **50ms**(1÷20초).
- 용어 재발: "Retry-After 정책이 들고 있다가 재시도" — 같은 날 정밀화한 걸("Retry-After는 B 전용") 마무리 정리에서 다시 섞음 → 인출 질문으로 즉시 교정, **blocking / bounded wait** 명칭 확보.

## 한 문장 봉인
> 거부 정책은 **대기 시간 × 동시 대기자 수**로 고른다 — 무계 손님(인바운드)은 fail-fast+Retry-After로 끊고, 유계 워커(아웃바운드)는 **bounded wait**(기다리되 마감, 마감 넘으면 거절 폴백)로 붙잡는다. 그 대기의 바닥엔 check-then-act 틈이 있고, 원자 연산으론 틈을 못 닫아 **synchronized**로 묶음 전체를 감싸되 — **잠은 문 밖에서** 자고, 깨면 도둑맞았을 수 있으니 while로 재확인한다(herd 미니어처).

## 학습법 회고

### 잘된 것
- **AtomicLong 후보 탈락법이 이 사이클의 백미.** 학습자가 대안을 던졌을 때 정답("안 됨") 대신 같은 시나리오를 그 코드에 재실행시켜 "깨진 게 ① 안인가, ①-② 사이인가"로 스스로 폐기하게 함 — race의 위치를 정확히 짚는 부수 효과까지.
- **학습자 주도 개선 2건이 코드에 실제 반영됨**: ① retryAfterSeconds 미재사용 이유를 먼저 질문 → 20배 과잉 수면 계산 → "나노 반환 메서드를 만들고 초 버전이 쓰게 하자" 리팩터링 자가 제안. ② "블로킹 의미 네이밍" 제안 → 관례(tryAcquire(timeout)) 학습 후 오버로드 채택.
- 퀴즈 Q1에서 답 대신 **조건을 되물은 것**("요청 한도가 몇인데?")을 정답으로 인정 — 축(대기자 수)의 암묵 적용이자 인출 형태의 진화.

### 메타 (이전 "바꿀 것" 적용 평가)
- **log_50 #1 ("출발점 한 줄을 학습자가 직접"): 부분 적용 ~50%.** 이 사이클은 arc 중간 이어받기라 "첫 응답" 조건 자체가 미성립. 실질 기회였던 "옵션 1 vs 2 감이 안 와" 순간에, 코치가 *어느 축이 흐릿한지* 되묻지 않고 두 옵션 의사코드를 직접 깔았다. 주제 선언은 학습자가 했으나 **축 지정 권한은 코치가 가져감**. 잘 굴러갔지만 아래 "바꿀 것"의 문제의식(주체성)과 같은 뿌리.

### 아쉬운 점 (방식 — 학습자 발의)
- **synchronized가 완성 코드로 등장했다.** 구현 전에 "어떻게 구현할지" 학습자에게 먼저 묻지 않아, 학습자가 구조화의 주체가 아니라 리뷰어가 됨 — "끌려가는 느낌, 주체성이 없으면 재미가 떨어진다." (학습 목적 = 스스로 구조화하는 재미, 그 뿌리를 정확히 건드린 아쉬움.)

### 바꿀 것 (다음) — 학습자 발의, 에이전트 초안 흡수
1. **문제 상황 제시 → 학습자가 먼저 푼다.** 새 개념이 필요한 구현에서 코치는: (a) *문제 상황만* 제시하고, (b) 학습자가 **개념 수준의 해법**을 말할 때까지 반복한다(틀리면 힌트로 다시, 맞을 때까지 — 정확히 짚으면 그때 넘어감), (c) 개념이 맞는데 용어를 모르면 그때 **이름을 붙여주고**, (d) 그것을 코드로 옮기는 **문법/구문은 코치가 공급**한다. 행동: 완성 코드를 먼저 보여줄 뻔한 순간마다 "이 문제, 어떻게 막을래?"로 바꿔 던진다. — log_50 항목(출발점 권한)의 일반화 이월(출발점→구현 주도권). 에이전트 초안("막힘 선언 시 흐릿한 축을 학습자가 지정")은 (b) 단계의 하위 규칙으로 흡수.
- 관찰 항목(추가 아님): 용어 재발(Retry-After) 1회 — 패턴인지 다음 사이클에서만 관찰.
